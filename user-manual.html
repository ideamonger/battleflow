Battleflow User Manual

Minimum viable wargame simulation engine for Operation IRON TIDE.

1. Purpose

Battleflow is a browser-based wargame simulation engine for exploring Blue and Red courses of action against terrain, movement, weapons, readiness, morale, supply, and objective-control rules. The current build is an MVP seeded with the Operation IRON TIDE scenario and a transparent JSON ontology.

The engine is intended for experimentation, planning support, and scenario analysis. It is not an authoritative combat model and does not encode classified performance data.

2. Quick Start

  1. Open the Battleflow site.
  2. Review the map, objectives, and Blue/Red starting areas.
  3. At T+0, drag units within their feasible starting areas if you want to adjust placement.
  4. Select a unit in the force tree or on the map.
  5. Use the Orders panel to issue a destination, timing, speed, task, and failure instruction.
  6. Run the simulation with Step, Play, or Batch.
  7. Use the COA panel to compare expected value, worst-case floor, regret, risk, and score.
  8. Export the state JSON when you want to preserve a run.

4. Terrain and Map

The default map is a 32 km by 18 km local grid. Terrain is synthesized from human-readable zones into one-kilometer cells. Each cell records a domain, movement effect, combat modifier, altitude, soil/accessibility/use data for land, water conditions for littoral zones, and global air conditions.

Terrain effects are deliberately simple. Beaches slow movement and penalize attacks, high ground improves attack effects, port and built areas impose urban friction, and water/air domains use lightweight weather and sea-state modifiers.

5. Forces and Ontology

Units are organized hierarchically with parent identifiers. A brigade may contain battalions, battalions may contain companies, and companies may contain platoons or specialist elements. The engine only needs consistent IDs and parent links.

Unit field Meaning
side Blue or Red.
echelon Command level such as platoon, company, battery, battalion, squadron, or brigade.
domain land, water/sea, or air.
location Grid location, frontage, depth, and altitude.
capabilities Role, personnel, combat power, firepower, mobility, readiness, morale, supplies, and weapons.
weapons Weapon name, range, rate of fire, destructive power, and ammunition.
orders Task, start time, arrival time, destination, speed, direction, and failure behavior.

6. Positioning and Starting Areas

The scenario defines separate feasible starting areas for Blue and Red. At T+0, units can be dragged on the map as long as they remain inside their side's starting area. Once the simulation has advanced, movement should happen through orders rather than direct drag placement.

The Forces tab validates that initial placements are legal and that Blue and Red starting areas do not overlap.

7. Orders

Orders are issued to units at the configured command level or above. The current scenario uses company as the minimum command echelon. Each order answers:

  • Where the unit should go, in local grid coordinates.
  • When the order starts.
  • When the unit should arrive.
  • What task it should perform.
  • What it should do if the order fails by the arrival time.

Active orders draw dashed paths on the map. If a unit misses its arrival timing, the failure instruction is logged as a branch event.

8. Simulation Modes

Mode Behavior
Step Advances by the scenario time step, currently 10 simulated minutes.
Real time Advances slowly for observation and teaching.
Accelerated Runs the simulation at a faster interactive pace.
Fast Advances larger increments for quick exploration.
Batch Runs the full scenario horizon as fast as possible, then presents the final state.

9. Adjudication Rules

The MVP uses deterministic, transparent adjudication rather than hidden stochastic resolution. Movement depends on unit mobility, readiness, morale, supply, terrain, air weather, or sea state. Engagements depend on weapon range, destructive power, rate of fire, ammunition, attacker readiness/morale, target cover, and terrain modifiers.

The engine updates strength, readiness, morale, ammunition, fuel, sustainment, objective control, and event logs at each time step.

10. COA Scoring

The COA panel compares the seeded courses of action against four Red responses: likely, most dangerous, deceptive, and adaptive. Each COA receives:

  • Expected value across Red responses.
  • Worst-case floor for maximin reasoning.
  • Worst regret for minimax-regret reasoning.
  • Risk adjustment.
  • Composite score.

In the default Operation IRON TIDE setup, COA 3 is designed as the hedged minimax-regret baseline, COA 2 as the maximin lodgment-first posture, and COA 1 as the opportunity-seeking high-tempo thrust.

11. Loading and Exporting Data

Use Load JSON to import another Battleflow scenario. Use Export to save the current simulation state, including clock, metrics, terrain grid, objectives, units, and event log.

The Data tab can also apply edited scenario JSON directly. For reliable results, keep unit IDs, parent links, objective IDs, terrain bounds, and command echelons consistent.

12. Optional AI Order Parsing

The AI tab includes a local rule-based parser and an optional OpenAI Responses API parser. The local parser works without network access. The API parser requires an API key and should only be used in trusted local or internal testing environments because browser-side API keys are visible to the page user.

13. Troubleshooting

Symptom Likely cause Fix
Scenario does not load The page was opened as a local file or JSON is invalid. Serve the folder over HTTP/HTTPS and validate the JSON.
Unit cannot be dragged The simulation has advanced or the target is outside the starting area. Reset to T+0 or drag within the correct feasible starting area.
Order is rejected The unit is below the command echelon minimum or fields are invalid. Select a commandable unit and check coordinates, timing, and speed.
OpenAI parsing fails Missing API key, blocked network, or API error. Use Parse Local or configure a secure server-side proxy for public deployments.

14. Limits

Battleflow MVP is intentionally simple. It does not model classified system performance, detailed logistics networks, stochastic sensor uncertainty, command latency, full air tasking, naval maneuver, civilian movement, legal review workflows, or validated casualty prediction. Its value is transparency, rapid iteration, and explainable experimentation.
